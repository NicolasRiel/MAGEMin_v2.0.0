# G_excess (mixing-model) comparison: MELTS vs MAGEMin

Companion to `gbase_verification/` (which checked pure-endmember standard-state G0). This
checks **G_excess** - everything beyond the weighted sum of pure-endmember gbase: ideal
configurational entropy + Margules/order-parameter excess energy - for every gh solution
phase, calling real `xMELTS-master` source functions directly (not MAGEMin's CLI output).

## Methodology

- **Real MELTS side**: each phase's own `gmixXXX(mask, t, p, r, gmix, ...)` function is called
  directly from a small standalone C harness linked against `xMELTS-master`'s compiled
  `gibbs2.o` + `libMELTSbatch.a`. This returns the full molar Gibbs energy of mixing (ideal
  entropy + Margules/order-parameter excess), with order parameter(s) solved internally by
  each file's own `order()`/`pureOrder()` Newton iteration - the harness only ever supplies
  the reduced composition `r[]`.
- **gh side**: `gh_objective_functions.c`'s `obj_gh_XXX(n, x, grad, SS_ref_db)` functions are
  called completely unmodified, with a hand-populated minimal `SS_ref` struct. Setting
  `gb_lvl[i]=0` for every endmember collapses gh's own `df_raw = sum(p_i*(gb[i]+mu_Gex[i])) +
  Sconfig` construction down to exactly `Gex_total + Sconfig` - the same quantity real
  `gmixXXX` reports - with `ape[i]=1`/`fbc=1.0` forcing `factor=1.0` exactly. No production
  code was modified for this verification pass.
- `R = 0.0083144` kJ/mol/K is used throughout (MAGEMin's own convention, matching `gb_lvl`/
  `mu_Gex` already being in kJ), not the raw 8.3143 J/mol/K.
- Composition-basis conversions between gh's plain `p[]` (mole fractions, gh's own `EM_list`
  order) and real MELTS' `r[]` (which varies by phase - either a "drop the dependent
  endmember" subset, or for cpx/opx a genuine linear recombination) were derived directly
  from each source file's own site-fraction/`FR`-macro definitions, not assumed.
- ol/bi/fsp/hb/lc were swept over the full 10x10 T-P grid (700-1600 degC, 1-25 kbar, same
  grid as `gbase_verification/`) x 2 test compositions (equal-mixture, 0.7-dominant
  asymmetric split). The remaining phases (order-parameter and linear-recombination-basis
  models, plus liq) were spot-checked at 3 T-P points (700C/1kbar, 1000C/10kbar,
  1400C/20kbar) x 1-2 representative interior compositions each, given the much larger
  per-phase harness effort (embedded order-parameter solves, per-calibration W-tables) -
  not a full grid+composition sweep for every phase/calibration.
- Only xMELTS (`EM_database==0`) was tested for phases whose mixing model doesn't vary by
  calibration (everything except liq and cpx's SS2 term, which is already known to be
  xMELTS-only). liq was tested against `gmixLiq_v34`/`MODE__MELTS` (real MELTS' actual
  dispatch target for gh's ported W-table, confirmed via gh's own header comment citing
  `param_struct_data_v34.h`), not the newer NA=19/NS=1 CO2-CaCO3-speciation branch, which
  neither real xMELTS' v34 liquid nor gh's port implement.

## Summary of findings

**Three real bugs found in mel/rhm/spn (mel, rhm, spn) - ALL FIXED 2026-07-14**, all sharing
the exact same root cause: `obj_gh_mel`/`obj_gh_rhm`/`obj_gh_spn` computed the full
joint-ordering `G(r,s*) = H(r,s*) - T*S(r,s*)` as their reported "Gex", but never subtracted
`ENDMEMBERS(r)` (the composition-weighted sum of each endmember's own individual
intracrystalline-ordering correction) the way real MELTS' `gmixMel`/`gmixMsg`/`gmixSpn` all do
(`*gmix = G; ... *gmix -= ENDMEMBERS;`). Since gh's `gbase[]` table *already* carries each
endmember's own ordering correction separately for rhm (added via `GH_rhm_pure_order_G` in
`GH_gem_function.c`) - but NOT for mel/spn, whose gbase[] carries no correction at all - the
un-subtracted `G(r,s*)` meant rhm's production total silently **double-counted** each
endmember's correction, while mel/spn's simply lacked it from the total entirely (same net
effect: Gex not properly reduced to a true Gibbs-energy-of-mixing quantity). Proven two ways
per phase before fixing: (1) the reported "Gex" failed to vanish at a pure-endmember
composition (a defining property of any correct Gibbs-energy-of-mixing function), and (2) for
rhm and spn, `gh_reported = real_reported + ENDMEMBERS(r)` matched to 5+ significant figures
once `ENDMEMBERS(r)` was computed independently from a from-scratch replica of each phase's
own pure-endmember ordering macros. Effect size scaled with how large each phase's individual
endmember ordering corrections happened to be: tiny for rhm (~0.5-0.9 kJ), large for mel
(~2.8-4.9 kJ, entirely from gehlenite's own correction) and **very large for spn (~30-90
kJ)**, since spinel's individual endmember ordering corrections (SP_G, HC_G, MT_G, CR_G,
UV_G) are all tens of kJ. cpx (which has the exact same architecture) was checked and was NOT
affected - it already correctly did `Gex = Graw - Emix` (`Emix = p5*es_end`, only essenite's
correction is nonzero for cpx) - this pattern is what the 3 fixes now also follow.

**Fix**: each of `obj_gh_mel`/`obj_gh_rhm`/`obj_gh_spn` now computes `ENDMEMBERS(r)` (reusing
rhm's existing gbase-construction corrections for rhm; newly porting gehlenite's `GH_G`
Landau correction for mel and all 5 of spn's `CR_G`/`HC_G`/`MT_G`/`SP_G`/`UV_G` corrections,
via new file-local `GH_rhm_mix_endmember_order_G`/`GH_mel_mix_geh_order_G`/
`GH_spn_mix_endmember_G` helper functions in `gh_objective_functions.c`, matching this file's
existing convention of duplicating small helpers per-function rather than sharing statics
across files) and subtracts it from Gex, with the analytic gradient (`dGex[]`) adjusted to
match (`d(ENDMEMBERS)/dp_i` is simply that endmember's own correction, since it's linear in
composition and T,P-only). Verified post-fix against real MELTS: rhm and spn now match to
noise level (~0.0003-0.003 kJ, down from ~0.4-1.7 kJ and ~30-90 kJ respectively) across the
full T-P range and multiple compositions, including finite-difference gradient checks. mel
matches to noise level (~0.0003 kJ) at symmetric/near-pure compositions but retains a real,
separate, pre-existing residual (~0.02-0.16 kJ) at genuinely asymmetric compositions - traced
to real MELTS' own joint order-parameter solve using an **unclamped, undamped Newton
iteration with its bound-checks literally commented out in the source** (`sources/
melilite.c`'s `order()`), a different, more fragile algorithm than gh's deliberate bisection
choice (already documented in gh's own header comment as intentional, since real's Newton can
fail to converge/cycle at compositions with negative-curvature W22p regions) - this is a
separate, pre-existing solver-fidelity trade-off, not part of the ENDMEMBERS bug just fixed,
and out of scope for this pass. Full MAGEMin build verified clean and sanity-tested (xMELTS/
rMELTS/pMELTS all converge normally, rhm/spn/cpx phases appear in realistic assemblages, no
regressions) after all 3 fixes.

**opx: was a cpx wrapper (~9-10 kJ off), now a full independent port - FIXED 2026-07-14**: gh's
`obj_gh_opx` used to just call `obj_gh_cpx` verbatim. Real `orthopyroxene.c`'s own `isClino()`
unconditionally returns `FALSE` (dead `#ifdef ISCLINO`/live `#else` fallback, matching
`clinopyroxene.c`'s unconditional `TRUE` the same way), and since real cpx/opx are the exact
same source machinery parameterized by `clino`, this affects essentially all ~245 Taylor
coefficients in the mixing model (not just a handful of W-parameters). Mechanically
transcribed opx's own coefficient set via a Python script, cross-validated against gh's
already-known-correct cpx constants (0 mismatches). Found and fixed 2 real bugs along the way:
a missing `H7`/`S7`/`V7` macro (defined separately from `H1`..`H6` in the source, easy to
miss), and - the actual root cause of the persisting ~10 kJ gap - real `orthopyroxene.c`'s own
`pureOrder`/`purePyx` functions (computing the `ENDMEMBERS(r)` reference subtracted from the
mixing model) hardcode `clino=TRUE` **internally**, regardless of the surrounding opx context,
so real opx's endmember references reuse cpx's own values (only essenite nonzero, exactly
like cpx) rather than opx-specific ones. Post-fix, diffs dropped from ~9-10 kJ to consistently
sub-1 kJ across a broad composition/T/P sweep, matching this pass's general residual-noise
level. See the `opx` section below for full detail and data.

**liq (xMELTS): two real bugs found and FIXED 2026-07-14.** (1) 10 of 12 CO2-touching
W-table entries (of 78 total) were transcribed wrong (not rounding slips - e.g. SiO2-CO2 was
0.0, should be 66600.7; Fe2SiO4-CO2 was -32464.5, should be 134680.2), isolated by direct,
name-based diffing against real `param_struct_data_v34.h`'s self-labeled
`meltsModelParameters[]` table rather than trusting positional assumptions. (2) real
`gmixLiq_v34` adds an extra, deliberate ideal-entropy term
`R*t*(x[H2O]*log(x[H2O])+(1-x[H2O])*log(1-x[H2O]))` on top of its plain per-component sum,
applied specifically to H2O (always the last of its NA=19 components) - entirely missing from
gh's port. H2O is likewise always the last endmember in gh's own `EM_list` for every liq
calibration, so the fix (`p[n_em-1]`) is generic across all 3 calibrations, not
xMELTS-specific. Post-fix, xMELTS liq matches real MELTS to <0.001 kJ across a wide sweep of
compositions/T/P plus a finite-difference gradient check.

**liq (pMELTS): W-table was wholesale wrong, FIXED 2026-07-14.** gh's pMELTS W-table had been
built (in an earlier session) from xMELTS' own table with CO2's 12 touching entries deleted -
a fundamentally wrong assumption. Real pMELTS liq uses its own, separately-calibrated
66-entry `pMeltsModelParameters[]` table (same file); only 1 of 66 values (a coincidental
0.0) matched gh's prior table. Replaced wholesale with values extracted directly from source.
The missing-H2O-entropy-term fix above applies unchanged (H2O is index 11 of 12 in gh's own
pMELTS `EM_list`). Post-fix, matches real MELTS (`gmixLiq_v34`, `MODE_pMELTS`) to <0.001 kJ
across a wide compositional sweep including rich (10+ simultaneously-nonzero endmember)
compositions.

**liq (rMELTS): real, quantified, ~0-0.6 kJ gap - NOT fixed, documented as a known
architectural simplification.** Real rMELTS liq (`sources/liquid_CO2_H2O.c`'s
`gmixLiq_CO2_H2O`, dispatched for `calculationMode==MODE__MELTSandCO2_H2O`) is not the same
closed-form symmetric-Margules model as xMELTS/pMELTS - it solves one internal order
parameter via Newton iteration (`order()`, matrix LU decomposition) before evaluating a
Taylor-expansion energy (`fillG`), and its own W-table
(`meltsAndCO2_H2OModelParameters[]` in `param_struct_data_CO2_H2O.h`) includes a genuine
extra species (CaCO3) absent from xMELTS/pMELTS entirely. gh's current rMELTS liq reuses the
plain closed-form `obj_gh_liq` architecture (correctly noting 9 of 78 W-values differ from
xMELTS, but not that the underlying *model* differs). Numerically quantified via a working
harness (`InitComputeDataStruct()` + direct `gmixLiq_CO2_H2O` call): diff is near-zero
(~0.0002-0.001 kJ) for simple/trace compositions, growing to 0.26-0.59 kJ for rich
compositions with both CO2 and H2O present simultaneously at high T (900-1600°C) - real but
modest (roughly 1-2% relative to typical Gmix magnitudes of -10 to -60 kJ), consistent with
the missing physics being a second-order correction rather than a first-order error. Properly
fixing this would mean porting the order-parameter solve and the CaCO3 species - a
substantial new implementation, out of scope for this pass (deferred, same treatment as the
opx `isClino` gap was before it was fixed).

**All other phases verified correct** (diffs at or below floating-point/convergence-tolerance
noise, ~0.0001-0.0006 kJ): ol, bi, fsp, hb, lc (full grid), cum, nph, kls (spot-checked), cpx
(spot-checked).

### opx - Orthopyroxene - **REAL BUG: was a cpx wrapper, now a full independent port - FIXED 2026-07-14**

Was previously confirmed as an "expected mismatch, not new" (gh's `obj_gh_opx` just called
`obj_gh_cpx` verbatim, ~9 kJ off, reasoning both source files hardcode clino=TRUE). On
revisiting: real `orthopyroxene.c`'s own `isClino()` unconditionally returns `FALSE` (dead
`#ifdef ISCLINO`/live `#else` fallback, confirmed by direct reading), and since real cpx and
opx are the SAME source-code machinery parameterized by `clino`, this affects essentially all
~245 Taylor coefficients in the mixing model, not just a handful of W-parameters (real
clinopyroxene.c ALSO zeroes 7 "vertex" corrections H1..H7/S1..S7/V1..V7 when clino=TRUE and
activates a separate "sliding pigeonite standard state" pH1..pH7 instead - opx reverses this).

**Fix**: mechanically transcribed all ~244 opx-specific (clino=FALSE) Taylor coefficients via
a Python script (`cpx_opx_taylor.py`) that evaluates every relevant `#define` macro from
`clinopyroxene.c` (text-identical to `orthopyroxene.c`) numerically - cross-validated first
by running with clino=TRUE and confirming all outputs matched gh's own already-verified cpx
constants exactly (0 mismatches of 244, later 710 once cross-checked against the actual pasted
C code). Two real bugs found and fixed during this process:
1. **Missing `H7`/`S7`/`V7`** - defined separately in the source (not in the same contiguous
   comment block as `H1`..`H6`), initially missed. Real but this specific omission turned out
   to be `Gex`-invariant at the composition first used to detect the ~10 kJ residual (it
   shifts `Graw` and `EN_G` by the exact same amount), so fixing it alone didn't close the gap
   - but it's a genuine, necessary correction (affects the gradient, and other compositions).
2. **The actual root cause**: real `orthopyroxene.c`'s own `pureOrder`/`purePyx` functions
   (computing the `ENDMEMBERS(r)` reference values subtracted from the mixing model) hardcode
   `static const int clino = TRUE;` **internally**, regardless of the surrounding opx
   (clino=FALSE) context - confirmed by direct reading of `orthopyroxene.c` lines 1385, 1470.
   This means real opx's endmember reference corrections use CPX's own Taylor coefficients,
   not opx's - so `DI_G=EN_G=HD_G=CA_G=CF_G=JD_G=0` for opx too (matching cpx exactly), and
   only essenite is nonzero, using the exact same `GH_cpx_pure_ES_G` already used by
   `obj_gh_cpx` (no separate opx version needed). The initial opx port used a from-scratch
   7-endmember `GH_opx_ends` (opx Taylor coefficients throughout), reasoning the whole
   `ENDMEMBERS` calculation should follow the overall clino=FALSE context - that was the
   actual bug, not caught until an exact-pure-endmember test (`p1=1`, cen) showed gh giving
   exactly 0 while real gave a genuinely nonzero -2.044 kJ, prompting a direct re-read of
   `purePyx`'s own source. `GH_opx_ends` is kept in the source (unused, `-Wunused-function`)
   as a documented dead end rather than deleted.

Post-fix, diffs dropped from ~9-10 kJ to consistently sub-1 kJ across a wide sweep of
interior compositions, T, and P (matching the residual-noise level seen throughout the rest
of this verification pass) - verified via value comparison across many T/P/compositions and

### liq - Liquid (rMELTS, `MODE__MELTSandCO2_H2O`) - **real, quantified gap - NOT fixed, documented as known simplification**
